# データモデル設計

## 1. 概要

本システムは、CNNによる欠陥検出モデルの推論結果を管理するだけでなく、SaaS型AIサービスとしての運用・分析・品質改善・再学習データ生成を目的としたデータモデルを設計する。

推論結果を中心に、ユーザー、アップロード画像、APIログ、エラーログ、ヒューマンレビュー、再学習候補データを関連付けることで、運用管理とモデル改善の両方を支援する構造とした。

---

## 2. 設計方針

- データの正規化による冗長性の排除
- 推論結果を中心とした構造設計
- ログおよび障害追跡の分離
- レビューおよび再学習フローのデータ化
- 将来的な拡張性の確保
- 運用監視と分析を考慮したカラム設計

---

## 3. エンティティ構成

### users
ユーザー情報を管理する。ログイン識別子、権限、アカウント状態などを保持する。

### uploaded_images
アップロードされた画像のメタ情報を管理する。ファイル名、保存パス、MIMEタイプ、ファイルサイズ、ハッシュ値、画像サイズなどを保持する。

### predictions
AI推論結果を管理する中核テーブル。対象画像、実行ユーザー、推論ラベル、信頼度、モデルバージョン、失敗理由などを保持する。

### review_queue
低信頼または検証対象の予測を管理する。レビュー状態、人手ラベル、AI補助レビュー内容を保持する。

### dataset_candidates
再学習に使用するデータ候補を管理する。候補理由や承認状態を保持する。

### request_logs
APIリクエスト履歴を記録する。リクエストID、エンドポイント、HTTPメソッド、ステータスコード、処理時間などを保持する。

### error_logs
エラーおよび障害情報を記録する。エラーコード、エラーメッセージ、詳細情報を保持する。

---

## 4. リレーション

- users : uploaded_images = 1:N
- users : predictions = 1:N
- uploaded_images : predictions = 1:N
- predictions : review_queue = 1:0..1
- predictions : dataset_candidates = 1:0..1
- users : request_logs = 1:N
- predictions : request_logs = 1:N
- request_logs : error_logs = 1:N

---

## 5. 主な設計ポイント

### 5.1 トレーサビリティ
ユーザー、画像、推論結果、リクエストログを関連付けることで、どの入力からどの出力が生成されたか追跡可能にする。

### 5.2 品質改善ループ
低信頼推論や誤分類の可能性があるケースを `review_queue` に登録し、レビュー後に `dataset_candidates` に連携することで、モデル改善サイクルを支援する。

### 5.3 運用監視
`request_logs` と `error_logs` を分離することで、正常系と異常系の分析をしやすくし、応答時間や障害発生傾向も可視化できるようにする。

### 5.4 拡張性
AIレビュー要約、AIレビュー推奨、類似ケース群、モデルバージョンなどを保持できるようにし、将来的な分析・機能追加に対応できる構造とする。

---

## 6. 拡張フィールド

将来的な分析およびAI補助機能を考慮し、以下のフィールドを含める。

- `model_version`
- `is_low_confidence`
- `failure_reason`
- `similar_case_group`
- `ai_review_summary`
- `ai_review_recommendation`
- `candidate_reason`
- `candidate_status`
- `latency_ms`
- `error_code`
